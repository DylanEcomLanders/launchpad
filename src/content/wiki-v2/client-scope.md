Every project has a defined scope. If it's not in the document, it's not in scope. This protects both us and the client.

> **Every hour spent on scope management saves five hours of scope creep.** It feels bureaucratic in the moment. It prevents the "this project was supposed to take 40 hours and we're at 80" disasters that kill profitability and team morale.

## What's typically in scope

For a standard landing page:

- Custom design (desktop + mobile) based on the brief
- Development build on Shopify
- 2 rounds of design revisions
- 1 round of development revisions
- Responsive across standard breakpoints
- Basic QA and browser testing
- Go-live deployment
- 1 week post-launch monitoring

## Typically out of scope (unless quoted)

- Copywriting (we build with client-provided copy)
- Photography or stock image sourcing
- SEO strategy or keyword research
- Email marketing setup
- Third-party app configuration
- Ongoing maintenance after launch
- Additional pages beyond what's quoted
- Hosting or domain management
- Logo design or brand identity work
- Custom Shopify app development

## Change request process

When a client asks for something outside the original scope, follow this every time.

| Step | Action |
| --- | --- |
| **1. Acknowledge** | Don't say "no" — say "absolutely, let me scope that out for you." |
| **2. Document** | Write down what, why it's outside scope, impact on timeline and budget. |
| **3. Estimate** | Hours by discipline, total cost, timeline impact. Confirm existing deliverables aren't affected. |
| **4. Get written approval** | Email saying "yes, go ahead" is enough. Verbal "sure" on a call is not. |
| **5. Update the project** | Add change request tasks to the board, update timeline if affected, send additional invoice. |

### Acknowledgement script

```
Thanks for the feedback. That's a great idea. It falls outside
the current project scope, so let me put together a quick estimate
for adding it. I'll have that to you by [date].
```

### Estimate format

```
Change Request: Add an interactive product comparison table to the landing page

Scope Impact:
- Design: +4 hours (table design, mobile layout, interaction states)
- Development: +6 hours (dynamic table, variant switching, responsive)
- QA: +2 hours

Total: 12 additional hours @ [rate] = £[amount]
Timeline Impact: +2-3 business days

This does not affect the existing deliverables or their timeline.
```

## Scope creep patterns

Learn to recognise these before they spiral.

### "While you're in there..."

> **What they say:** "Since you're already working on the hero, could you also update the navigation?"

Free work request disguised as a minor addition.
**Response:** "Happy to look at the navigation, that would be a separate task. Want me to estimate it?"

### "This should be quick..."

> **What they say:** "Can you just quickly add a pop-up? It shouldn't take long."

They're minimising the effort to avoid paying for it.
**Response:** "Pop-ups involve design, development, targeting logic, and testing. It's actually a solid half-day of work. Let me scope it properly so we do it right."

### "I thought this was included..."

> **What they say:** "I assumed the email capture form setup was part of the landing page."

Genuine misunderstanding (sometimes) or boundary testing (sometimes).
**Response:** "I can see why you'd expect that. Looking at the proposal, the scope covers [X, Y, Z]. Email integration is something we can definitely add — here's what that would look like."

### The drip feed

Small requests every few days that individually seem trivial but collectively add 15+ hours.
**Response:** Log every request. End of week, review the total. If it's over scope, have the direct conversation: "This week we've received 6 additional requests totalling about 8 hours of work outside the project scope. I want to make sure we handle these properly — want me to bundle them into a change request?"

### Feature creep during review

Client feedback that isn't revisions — it's new features.
- "Can we add a countdown timer?" = new feature
- "Can the countdown timer be bigger?" = revision (if it was in the design)

**Response:** Separate revision feedback from new feature requests. Process revisions normally. Run new features through the change request process.

## How to push back

You're not saying no. You're protecting the project, the timeline, and the quality of the deliverable. Frame it that way.

### On scope creep

```
I want to make sure we deliver the best possible [landing page]
within the timeline we agreed on. Adding [new feature] would
require an additional [X] hours and could push the launch date
by [Y] days.

I'd recommend we launch with the current scope and add
[new feature] as a Phase 2 improvement. That way we hit the
launch date and we can measure performance before adding
more complexity.

What do you think?
```

### On bad feedback

```
I appreciate the feedback. Before we implement that change,
I want to flag that [moving the testimonials below the fold /
removing the CTA from the hero / etc.] could negatively impact
conversions because [specific reason backed by data or best practice].

My recommendation would be to [alternative approach].
That achieves what you're looking for while maintaining the
conversion-focused structure.

Happy to discuss on a quick call if helpful.
```

### On timeline pressure

```
I understand the urgency. To hit [date], we'd need to reduce
the scope to [specific reduced scope]. Alternatively, we can
deliver the full scope by [realistic date].

I'd rather give you something excellent a few days later than
something rushed on time. Which would you prefer?
```

## When to re-scope

Sometimes the scope genuinely needs to change.

- Total change requests exceed 25% of the original project value
- The project direction has fundamentally changed since kickoff
- New requirements make the original timeline impossible
- The client and team are both frustrated with constant change requests

**Re-scoping process:**

1. **Pause and assess.** Stop work temporarily if needed.
2. **Document everything.** Original scope, all changes requested, current state.
3. **Honest conversation.** "The project has evolved significantly from the original brief. I think we should re-scope to make sure the plan, budget, and timeline all align with where we are now."
4. **New proposal.** Treat it as a fresh scope document. Credit any work already completed.
5. **Get it signed.** New scope = new agreement. Don't proceed on a handshake.

## Contract clauses

Standard language to include in every project agreement.

**Scope:**

```
The scope of this project is limited to the deliverables described
in Section [X] of this agreement. Any work not explicitly listed
is considered out of scope and will require a separate change
request and written approval before work begins.
```

**Revisions:**

```
This project includes [2] rounds of design revisions and [1] round
of development revisions. A "revision" is defined as changes to
existing deliverables, not requests for new features or additional
pages. Additional revision rounds will be billed at our standard
hourly rate of £[rate]/hour.
```

**Change requests:**

```
Any changes to the scope, timeline, or deliverables described in
this agreement will be documented as a Change Request. Change Requests
will include an estimate of additional time and cost. Work on Change
Requests will not begin until the client provides written approval
(email confirmation is sufficient).
```

**Timeline:**

```
The estimated project timeline is [X] weeks from kickoff, subject to
timely client feedback (within [2] business days of each deliverable).
Delays in client feedback will shift the project timeline by an
equivalent number of business days.
```
