# Flow: Conversion Engine

## When This Flow Applies

This is the full offer — team-led roadmap, ongoing conversion system management. It's not a linear project with a start and end. It's a sprint-based loop that compounds every month.

The Conversion Engine is essentially D&D+CRO running on a repeating sprint cycle, with a strategic roadmap layer on top. Each sprint produces conversion assets. Each asset generates data. Each data point informs the next sprint.

> **Full offer details** (pricing, sales process, positioning, partnership economics) live in the admin wiki.

---

## The Structure

```
INBOX → Onboarding Sprint → [ SPRINT LOOP ] → Ongoing
                                  ↑        ↓
                              Roadmap ← Measure
```

The partnership has three layers that run simultaneously:

1. **Roadmap Layer** — The 60-90 day visual roadmap (Miro). Updated monthly based on data.
2. **Sprint Layer** — Monthly build sprints. Each sprint = research → design → build → launch → measure.
3. **Reporting Layer** — Monthly reports proving ROI. Revenue attribution, CVR movement, test results.

---

## Onboarding Sprint (Month 1)

Month 1 is unique — it's a setup sprint, not a build sprint. The goal is: full audit, visual roadmap, quick wins live, system ready for ongoing sprints.

### Week 1: Access & Deep Audit

**Actions:**
- Complete all Onboarding Inbox requirements (access, assets, details)
- Run the full Conversion Matrix with backend data (GA4, Shopify, heatmaps)
- Heatmap analysis on top 5 highest-traffic pages
- Funnel drop-off mapping: landing → PDP → cart → checkout → purchase
- Competitor benchmark (top 3 competitors' post-click experiences)
- AOV opportunity mapping (bundles, upsells, post-purchase)
- Copy and offer audit

> **Reference:** [CRO Audit Process](/tools/ops-wiki?section=06-audit-process) for audit methodology, [Conversion Engine Wiki — Onboarding](/tools/conversion-engine?section=02-onboarding) for the full onboarding checklist.

### Week 1-2: Roadmap Build

**Actions:**
- Build the 60-90 day visual roadmap on Miro
- Prioritise all conversion assets by ICE score (Impact × Confidence × Ease)
- Sequence Month 1 / Month 2 / Month 3 priorities
- Create testing roadmap with if/then hypothesis chains
- Map AOV optimisation opportunities
- Identify and execute quick wins (fixes that don't need a full build)

**Roadmap structure:**
- Each item is a "conversion asset" — a specific page, flow, or optimisation
- Each asset has: hypothesis, success metric, estimated impact, ICE score
- Assets are grouped by type: Page Builds, Checkout Optimisation, AOV, Post-Purchase

### Week 2: Client Kickoff

**Actions:**
- Present audit findings — "Here's what we found and what it's costing you"
- Walk through the visual roadmap — "Here's our 90-day plan"
- Confirm Month 1 deliverables — "Here's what we're building first and why"
- Agree on communication rhythm (weekly async, monthly call)
- Ship any quick wins identified during audit

> **Reference:** [Client Communication & Updates](/tools/ops-wiki?section=14-client-comms) for kickoff structure.

### Week 3-4: First Build

Start the first conversion asset from the roadmap. This follows the standard D&D+CRO stages within the sprint:

```
CRO Brief → Design → Client Review → 🚧 HANDOFF → Build → 🚧 SELF-QA → QA → 🚧 LAUNCH PREP → Go-Live
```

Every gate from the D&D+CRO flow applies here. No shortcuts because it's a retainer.

## The Sprint Loop (Month 2+)

Each month follows the same cadence. The sprint loop is the engine.

### Week 1: Strategy & Planning

**Actions:**
- Review last month's test results and page performance
- Update the visual roadmap based on data (shift priorities if needed)
- Write CRO briefs for this month's conversion assets
- Write design briefs with section-by-section breakdown
- Async update to client: "Here's what we're building this month and why"

**Key principle:** We decide what gets built. The client approves the direction. If they want to override a priority, we explain why our recommendation should come first — backed by data.

### Week 2-3: Design & Build

**For each conversion asset in this sprint:**

1. **Design** — Desktop + mobile, CRO-informed, hypothesis-driven
2. **Client Review** — One round, structured feedback
3. **🚧 Design → Dev Handoff** — Full checklist, hard stop
4. **Build** — Section by section, test infrastructure if A/B
5. **🚧 Dev Self-QA** — Full checklist before internal review
6. **Internal QA** — Cross-browser, design fidelity, performance

> **References:** Same SOPs as D&D+CRO flow — [Design to Dev Handoff](/tools/ops-wiki?section=02-handoff), [QA Checklist](/tools/ops-wiki?section=12-qa-checklist), [Build Process](/tools/ops-wiki?section=03-build-process).

### Week 4: Launch & Measure

**Actions:**
- **🚧 Launch Prep** — Test method confirmed, baseline recorded, tracking verified
- Push live / start test
- Monitor for first 48 hours
- Begin collecting data for this month's report
- Plan next month's sprint priorities based on emerging data

> **Reference:** [Deployment & Go-Live](/tools/ops-wiki?section=05-deployment), [Testing Framework](/tools/ops-wiki?section=07-testing-framework)

## The Full Scope Per Sprint

Each sprint can include any combination of these conversion assets:

| Category | Examples |
|----------|----------|
| **Page Builds** | Landing pages, PDPs, homepage sections, collection pages |
| **Checkout** | Express checkout setup, form optimisation, trust signals |
| **AOV** | Bundle pages, cross-sell integration, free shipping threshold |
| **Post-Purchase** | Thank you page redesign, one-click upsell, review request flow |
| **Testing** | A/B tests on existing pages, multivariate experiments |
| **Optimisation** | Speed improvements, mobile UX fixes, copy refinement |

> **Reference:** [Conversion Engine Wiki — Delivery](/tools/conversion-engine?section=03-delivery) for the full scope of work breakdown.

The three gates apply to EVERY conversion asset in EVERY sprint. No exceptions because it's a retainer.

## Monthly Reporting

Every month the client receives a report. This is non-negotiable — it's what proves the ROI and justifies the partnership.

**Report includes:**

| Section | Content |
|---------|---------|
| Pages shipped | Before/after screenshots, what changed and why |
| Test results | Hypothesis vs outcome, statistical significance |
| CVR movement | Site-wide and per page/funnel stage |
| AOV movement | Average order value trends, upsell performance |
| Revenue attribution | Revenue directly attributed to our changes |
| Next month roadmap | What's coming next and why |
| Strategic notes | Observations, competitive shifts, recommendations |

> **Reference:** [Reporting & Analytics](/tools/ops-wiki?section=08-reporting) for report structure and data sources.

## Roadmap Management

The visual roadmap is a living document. It gets updated at the start of every sprint based on:

- What we learned from last month's tests
- New data from analytics
- Client business changes (new product launch, seasonal shift, campaign change)
- Competitive landscape shifts

**Roadmap update cadence:**
- **Weekly:** Small adjustments (reorder items within current sprint)
- **Monthly:** Major review (add/remove items, shift priorities for next sprint)
- **Quarterly:** Full strategic review with client (QBR for Pro tier)

**The roadmap is the artefact that proves we're not just building pages — we're running a system.** If a client ever asks "what are you doing for me?" point to the roadmap.

## Partnership Health Indicators

Monitor these to predict churn before it happens:

| Indicator | Healthy | Warning | Critical |
|-----------|---------|---------|----------|
| Client response time | < 24h on approvals | 2-3 days | > 5 days |
| Feedback quality | Specific, actionable | Vague, "looks good" | No feedback at all |
| Roadmap engagement | Asks questions, suggests ideas | Passive approval | Doesn't look at it |
| Meeting attendance | Shows up, engaged | Cancels occasionally | Ghosts meetings |
| Scope requests | Within framework | Occasional ad-hoc asks | Constant scope creep |

If you see 2+ warning indicators, have a proactive conversation: "I want to make sure we're delivering maximum value. Here's what I'm seeing — let's talk about how to get the most out of the partnership."

> **Reference:** [Scope Management & Change Requests](/tools/ops-wiki?section=15-scope-management) for handling scope creep within the partnership model.
