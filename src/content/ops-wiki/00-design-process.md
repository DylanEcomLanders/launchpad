# Design Process & Workflow

## Overview

Every landing page project at Ecom Landers follows the same design workflow regardless of scope. This keeps quality consistent, timelines predictable, and clients confident. No design work begins without a signed-off brief, and no design goes to dev without client approval.

## Design Request Flow

### 1. Brief Received
- Client submits brief via the onboarding portal or project kickoff form
- Account lead reviews for completeness (brand assets, goals, target audience, competitor references)
- If anything is missing, request it before assigning to a designer — do not start with gaps

### 2. Wireframe
- Low-fidelity layout in Figma (greyscale, no brand styling)
- Focus on information hierarchy, section order, and CTA placement
- Include mobile wireframe alongside desktop — never desktop only
- Share with client for structural feedback before moving to visual design

### 3. High-Fidelity Design
- Apply brand colours, typography, imagery, and tone
- Design desktop and mobile simultaneously — mobile is not an afterthought
- All copy must be real (no lorem ipsum in client-facing designs)
- Include hover states, micro-interactions, and any animated elements as annotations

### 4. Internal Review
- Peer review by another designer or the design lead
- Run through the Design Review Checklist (below)
- Fix issues before showing the client

### 5. Client Review & Approval
- Present via Loom walkthrough — never just drop a Figma link
- Walk through the page section by section, explaining design decisions
- Collect feedback in one round. If feedback is scattered, schedule a call
- Two revision rounds included in standard scope. Third round triggers a scope conversation

### 6. Approved — Handoff to Dev
- Follow the Design to Dev Handoff SOP (see `02-handoff.md`)

## Tool Stack

| Tool | Purpose |
|------|---------|
| Figma | All design work (wireframes, high-fi, prototyping) |
| Ecom Landers Component Library | Shared Figma library for reusable sections |
| Loom | Client walkthroughs and async feedback |
| Stark (Figma plugin) | Accessibility contrast checks |
| TinyPNG / Squoosh | Image compression before handoff |

## File Naming Convention

All Figma files follow this format:

```
[ClientName] — [ProjectType] — [Version]
```

Examples:
- `Hydra Supplements — Landing Page — v1`
- `Glow Skin — Homepage Redesign — v2`
- `Bolt Athletics — Collection Page — v1`

> **Never** name files "final", "final-v2", or "FINAL-REAL". Use version numbers.

## Design Review Checklist

Before any design goes to client review, confirm:

- [ ] Desktop and mobile designs are complete
- [ ] All copy is real (not placeholder)
- [ ] Brand colours and fonts match the style guide
- [ ] CTAs are prominent and above the fold on mobile
- [ ] Social proof sections use real reviews/testimonials
- [ ] Images are high quality and correctly cropped
- [ ] Spacing is consistent (use 8px grid)
- [ ] Accessibility: text contrast passes WCAG AA (4.5:1 minimum)
- [ ] Hover and interaction states are documented
- [ ] Page has a clear visual hierarchy — one primary CTA per viewport

## Timeline Standards

| Phase | Standard Timeline | Rush Timeline |
|-------|------------------|---------------|
| Wireframe | 1 business day | Same day |
| High-Fidelity Design | 2–3 business days | 1–2 business days |
| Internal Review | 0.5 business days | Same day |
| Client Review Round | 1–2 business days (depends on client) | 1 business day |
| Revisions per Round | 1 business day | Same day |
| Handoff Prep | 0.5 business days | Same day |

> **Total typical timeline from brief to approved design: 5–7 business days.** Rush projects can compress to 3 days but require the client to respond within hours, not days.

### What Causes Delays
- Missing brand assets at brief stage
- Placeholder copy that needs to be written
- Multiple stakeholders giving conflicting feedback
- Scope creep disguised as "small tweaks"

Flag these early. If you see them, raise it with the account lead the same day.
